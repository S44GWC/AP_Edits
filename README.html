# Trello-to-Sheets V2 Synchronisation Engine ## Overview This Google Apps Script project provides a one-way, automated synchronisation between a Trello board and a Google Workspace environment. It tracks the time cards spend in specific Trello lists (e.g., Editing, Proofing) and logs this data into a backend Google Sheet (`System Data`). This data is then formatted and visualised on a clean front-end sheet (`Production Dashboard`). The system uses a **Delta-Time Engine** and **In-Memory Batch Processing** to ensure high performance and avoid Google Apps Script execution timeouts. --- ## System Architecture The codebase is modularised into five distinct files to separate configuration, API logic, data processing, and front-end generation. ### 1. `Config.gs` The central hub for all static variables. * Contains the Trello API Key and Token. * Maps all Trello List IDs to readable constants. * Defines the names of the required Google Sheets and the array of `KNOWN_EDITORS`. * **Maintainer Note:** If the team changes Trello boards, or if new editors join the team, this is the *only* file that needs to be updated. ### 2. `TrelloAPI.gs` Handles all external requests to the Trello API. * `fetchCardsInList()`: Pulls all active cards for a given list. * `isCardCompleted()`: Checks if a card missing from the active pipeline has been moved to the 'Done' list or archived. * `extractCardMetadata()`: Parses the Trello card title (e.g., "School Name - Style"), description (for group numbers), and labels (to match against known editors). ### 3. `MainEngine.gs` The core logic script. **This is the only function that should be attached to a Time-Driven Trigger.** * Uses the **Delta-Time** method: Rather than relying on webhooks or exact timestamps, the script calculates the elapsed time (in milliseconds) between the current execution and the `Last Checked` timestamp on the sheet. * Uses **In-Memory Batch Processing**: It reads the entire sheet into a 2D array, performs all time additions and status updates in memory, and writes the array back to the sheet in a single `setValues()` call. This prevents API rate limits and Google Sheets cell-by-cell write timeouts. ### 4. `DashboardHelpers.gs` A collection of manual utility functions for database administration. * Includes tools to repair broken array formulas, migrate historical data from older system versions, and force-update the dashboard layout. * **Maintainer Note:** These functions do not run automatically. They are intended to be executed manually from the Apps Script Editor when maintenance is required. ### 5. `SheetSetup.gs` The initialisation script. * `buildProDashboard()`: Creates the `System Data` and `Production Dashboard` sheets from scratch. * It injects the required headers, sets up the `QUERY` formulas for the dashboard, and applies native duration formatting (`[hh]:mm:ss`) to ensure metrics are calculated accurately from raw milliseconds. --- ## Deployment & Setup Instructions If deploying this script to a new Google Sheet, follow these steps: 1. **Add the Code:** Open **Extensions > Apps Script** and create the five `.gs` files listed above, pasting in the respective code. 2. **Initialise the Sheets:** * Select `SheetSetup.gs`. * Choose the `buildProDashboard` function from the dropdown menu in the toolbar. * Click **Run**. Grant necessary permissions. This will generate your front-end and back-end sheets. 3. **Configure the Trigger:** * In the Apps Script editor, click the **Triggers** icon (the alarm clock on the left sidebar). * Click **Add Trigger**. * Set the function to run to `runSyncEngine`. * Set the event source to **Time-driven**. * Set the type to **Minutes timer** and choose an interval (e.g., **Every 5 minutes**). * Click **Save**. --- ## Developer Notes: The "Millisecond Trap" When working with durations in Google Sheets, be aware of how Sheets handles time maths. Google Sheets natively stores durations as **fractions of a day**. For example, 12 hours is stored as `0.5`. Because the Trello Engine calculates elapsed time in **milliseconds**, attempting to format a raw millisecond output (e.g., `43200000`) directly as `[hh]:mm:ss` in Sheets will break the display (Sheets will read it as 43 million days). **The Solution:** Whenever passing the backend millisecond data to a front-end view or a `QUERY` formula, the raw millisecond value must be divided by `86400000` (the number of milliseconds in a day). This converts it into a standard day-fraction, which can then safely be formatted using the native `[hh]:mm:ss` number format. This standardisation is handled automatically in the backend math columns (K, L, M) generated by `buildProDashboard()`.
